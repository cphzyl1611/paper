\section{Appendix}

\subsection{Related Work}
\label{app:related-work}

This section expands the related-work discussion in the introduction. Prior work provides several important pieces for APT defense evaluation: a common language for describing attacks, step-level observations of security products, knowledge bases of defensive techniques, and models for vulnerability severity or attack-path risk. These pieces are useful, but they do not use the same unit of analysis as this paper. Our framework scores an APT action together with its observed defense result and its first-failed victim-side control point. This unit is needed because a score should remain connected to a repairable control failure after it has been aggregated across actions, stages, and playbooks.

\subsection{Attack Knowledge Bases and Step-Level Product Evaluation}

MITRE ATT\&CK provides a common language for describing adversary behavior. Its tactics, techniques, and procedures are widely used in threat intelligence, red-team planning, adversary emulation, and security product evaluation~\cite{mitreAttack}. ATT\&CK is therefore a natural starting point for representing APT playbook actions.

For repair-oriented scoring, however, an ATT\&CK label is not enough. A technique label tells us what the attacker did, but it does not explain why the victim environment failed to stop the action. The same credential-related behavior, for example, may be enabled by weak password policy, missing multi-factor authentication, exposed remote login, insufficient credential storage protection, or successful phishing. These cases may share an attack label, but they do not point to the same repair action. Our framework therefore uses ATT\&CK-style action descriptions as input, while adding a separate attribution layer that maps each in-scope action to the victim-side control point that first failed.

MITRE ATT\&CK Evaluations move closer to defense assessment by reporting how products behave during adversary-emulation steps~\cite{mitreAttackEvaluations}. This step-level view is valuable because it makes product behavior visible under a shared emulation plan. In our setting, such observations are close to the raw evidence used for scoring: an action may be prevented, blocked, detected, or missed.

The remaining gap is how to turn step-level observations into an interpretable APT defense score. A step matrix can show where a product responded, but it does not decide which repairable control point should be charged for a missed action. It also does not directly distinguish the defensive value of blocking initial access from detecting activity near exfiltration. This paper builds on step-level observations, but adds control-point attribution, canonical phase weights, and stage aggregation semantics before computing scores.

\subsubsection{Defensive Knowledge Bases and Control Capabilities}

D3FEND complements ATT\&CK by organizing defensive techniques and their relations to adversary behaviors and digital artifacts~\cite{mitreD3FEND,kaloroumakis2021d3fend}. It is useful when the task is to reason about which defensive techniques may apply to an attack behavior.

This capability-centered view is related to our defense-technology crosswalk, but it is not the same as control-point attribution. D3FEND can indicate that a class of defensive techniques may counter an attack behavior. It does not decide, for a particular playbook action, which victim-side control point first failed and should receive the score loss. For example, a missed credential-dumping action may involve endpoint monitoring, credential storage protection, privilege control, or response failure. Treating all of them as possible countermeasures is helpful for capability reasoning, but the benchmark still needs a primary attribution for scoring and repair prioritization.

Our control-point attribution schema and defense-technology crosswalk use this narrower viewpoint. The attribution schema assigns each action to the first-failed control point used for scoring. The crosswalk then links that control point to the product categories expected to prevent, block, or compensate for the failure. They do not replace D3FEND; rather, they use a complementary viewpoint centered on scoring, failure attribution, and repair priority.

\subsubsection{Vulnerability Scoring and Attack-Path Analysis}

Vulnerability scoring and attack-path analysis address a different part of the security evaluation problem. CVSS provides a standardized way to describe and score software vulnerability severity~\cite{mell2007common,first2019cvss31}. Attack-graph methods, including MulVAL and related work, model how vulnerabilities, privileges, and configurations can be combined into multi-step attack paths~\cite{ou2005mulval,ou2006scalable}. Other studies assign quantitative or probabilistic metrics to attack graphs~\cite{wang2007measuring,wang2008probabilistic,frigault2008dynamic}, while cyber-risk quantification work estimates possible loss under different threat and control assumptions~\cite{erola2022system}.

These methods are valuable because they make severity, reachability, path risk, and expected loss explicit. Their unit of analysis, however, is usually a vulnerability, a configuration dependency, an attack path, or a loss distribution. This is different from evaluating how a deployed defense stack performs against action-level APT behaviors. A high vulnerability score does not say whether a defense blocks initial access, detects execution, prevents lateral movement, or only observes activity after compromise. Similarly, an attack graph can show that a path is reachable, but it does not directly convert action-level defense observations into phase-wise, control-domain, and product-category scores.

In this paper, the unit of analysis is an APT action paired with an observed defense result. The action is mapped to a victim-side control-point failure, assigned a canonical phase, linked to expected defense categories, and then aggregated through stage and playbook logic. In short, attack graphs mainly quantify attacker reachability or path risk, while our framework quantifies defense performance over a fixed set of APT playbooks.

\subsubsection{Position of This Work}

The distinction is not that prior work is unrelated, but that each line of work stops at a different point in the evaluation pipeline. ATT\&CK gives the attack language. ATT\&CK Evaluations provide step-level product observations. D3FEND organizes defensive techniques. CVSS and attack graphs reason about vulnerability severity, reachability, and risk. This paper connects these perspectives through control-point attribution and stage-aware scoring.

This positioning addresses the three limitations discussed in the introduction. First, it reduces black-box scoring by keeping score loss traceable to phases, actions, product categories, and control-point domains. Second, it addresses undifferentiated stage value by applying canonical phase weights and stage aggregation semantics, so that early blocking and late detection are not treated as equivalent. Third, it addresses missing attribution by linking missed or weakly handled actions to repairable victim-side control-point failures.
